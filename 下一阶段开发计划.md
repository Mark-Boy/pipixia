## 详细实施计划

### 阶段 1：前端真实 API 接入（P0-1，4h）

**目标**：将 5 个仍使用 mock 数据的页面接入真实后端 API

| 页面 | 当前状态 | 需要修改 |
|------|---------|---------|
| `overview-page.tsx` | 混合 mock + API | 移除 mock data，统一使用 `reportService` + `productService` |
| `products-page.tsx` | 混合 mock + API | 统一使用 `productService` |
| `audit-page.tsx` | 主要用 mock | 统一使用 `auditService` |
| `finance-page.tsx` | 主要用 mock | 统一使用 `reportService.getFinance()` |
| `risk-page.tsx` | 主要用 mock | 使用 `settingService.getRiskWords()` + 后端风控日志 |

**具体工作**：
1. 移除 `mock-data.ts` 中的 mock 数据引用
2. 确保所有页面使用 `@/services` 中的真实 API 调用
3. 处理 API 响应格式差异（前端 type vs 后端 DB model）
4. 添加错误边界和 loading skeleton

### 阶段 2：Celery Beat 定时任务（P0-2，2h）

**目标**：配置并验证 Celery Beat 调度器

当前 `worker/celery_app.py` 已有 `beat_schedule` 定义但未验证：
- `daily-profit-report` — 每日生成利润报告
- `exchange-rate-update` — 每小时更新汇率
- `stock-sync` — 每日库存同步
- `profit-circuit-breaker` — 每日利润熔断检查

**具体工作**：
1. 创建 `celery beat` 启动脚本
2. 验证定时任务是否正常触发
3. 添加任务执行日志追踪

### 阶段 3：利润熔断查询逻辑（P1-1，3h）

**目标**：实现 `profit_circuit_breaker` Celery 任务的完整逻辑

当前实现是占位符，需要：
1. 查询所有 active 商品
2. 使用实时汇率计算实际利润率
3. 低于阈值的商品自动标记为 `blocked`
4. 记录风控日志
5. 发送 Telegram 告警

### 阶段 4：订单同步业务逻辑（P1-2，4h）

**目标**：完善 Webhook 订单事件的完整处理链路

当前 `webhooks.py` 已接收事件但业务逻辑不完整：
1. 订单创建 → 创建本地 listing 记录
2. 订单发货 → 更新库存
3. 订单取消 → 释放库存
4. 订单完成 → 触发利润核算
5. 与 Shopee API 双向同步

### 阶段 5：测试补充（P2-5，3h）

**目标**：提升测试覆盖率到 80%+

新增测试：
1. 路由层测试（每个 endpoint 的 happy path + error path）
2. Schema 验证测试
3. Worker 任务单元测试
4. 前端 API 服务集成测试
